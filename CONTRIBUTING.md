# Contributing to EMERGENZ Biosecurity Intel Dashboard

This dashboard is a public health intelligence tool maintained by EMERGENZ Systems. Contributions that improve accuracy, accessibility, and clinical utility are welcome.

---

## Ground rules

- **Accuracy first.** Every factual claim must be traceable to a primary source such as WHO, CDC, ECDC, Africa CDC, PAHO, PHAC, USDA APHIS, WOAH, a peer-reviewed publication, or an official government release. Do not add information from secondary or opinion media as if it were authoritative.
- **Cite everything.** If you add or update a signal, timeline event, metric, or source, include the source URL and verification date in `src/data/signal-sources.json` and reference it by id.
- **No speculation.** This is an operational tool. Predictive content, opinion, or extrapolation beyond official statements must be clearly labelled or excluded.

---

## Getting started

```bash
git clone https://github.com/EMERGENZ-Corporation/biosecurity-intel-dashboard.git
cd biosecurity-intel-dashboard
npm install
cp .env.example .env.local
npm run dev
```

The MVP uses static, hand-curated signal data backed by primary sources.
No third-party API keys are required to build, validate, or run the
dashboard locally. Source-specific extractors can be added incrementally
in `scripts/` without changing the public `status.json` contract.

## What to contribute

| Area | How |
|------|-----|
| Signals | Edit `src/data/signals.json`. Every signal must declare a `primarySourceId` and a list of `sourceIds` that all exist in `signal-sources.json`. |
| Timeline events | Edit `src/data/signal-timeline.json`. Foreign keys to `signals.json` and `signal-sources.json` are enforced by `npm run validate:data`. |
| Sources | Edit `src/data/signal-sources.json`. Mark `primary: true` only for official authorities (WHO, CDC, ECDC, Africa CDC, PAHO, PHAC, USDA APHIS, WOAH, CDC NWSS, WastewaterSCAN, national MoH). Treat ProMED, CIDRAP, and Brown Pandemic Center as secondary. |
| Status contract | The public `/status.json` is regenerated by `npm run generate:status` from the signal data. Do not hand-edit `public/status.json`. |
| Bug fixes / UI | Open an issue first for significant changes. |

## Pull request checklist

- [ ] All new data has a source URL and verification date
- [ ] `npm run validate:data` passes
- [ ] `npm run generate:status` runs cleanly
- [ ] `npm run build` passes without errors
- [ ] `npm audit` reports 0 vulnerabilities (or documents the exception)
- [ ] Accessible: interactive elements have labels and focus states work
- [ ] PR description explains what changed and why, not just what

## Data update policy

Signal data should be updated only from primary sources or expert weekly reports cited as secondary. If a source changes or expires, open an issue rather than removing the reference silently.

Automated status refresh runs through `.github/workflows/update-data.yml` and `scripts/generate-status.mjs`. The production app is a static Vite build; it does not depend on serverless API routes at runtime.

The public `status.json` contract is published at `https://biosecurity-intel.emergenzsystems.org/status.json` (placeholder until deployed; the URL is configurable via the `PRODUCTION_URL` / `STATUS_URL` env vars in `.env.example` and the GitHub workflows). Use that file for uptime checks, freshness checks, and incident triage instead of scraping the dashboard UI.

## `public/api/v1/*` commit policy

The files in `public/api/v1/` (`signals.json`, `news.json`, `signal-sources.json`, `signal-timeline.json`, `feed.rss`) are **tracked in git** as a deliberate snapshot of the public API surface. They are regenerated from `src/data/*` by `scripts/generate-api.mjs`.

Rules:

1. **Never hand-edit `public/api/v1/*`.** Data flows one direction: `src/data/*` → `generate-api.mjs` → `public/api/v1/*`. Direct edits will be overwritten on the next workflow run.
2. **When you change underlying data** (`src/data/signals.json`, `signal-timeline.json`, `signal-sources.json`, or anything else that flows into the generator), run `npm run generate:api` and commit both the data change and the regenerated artifacts in the **same commit**. This keeps the diff reviewable and the deployed snapshot in sync with the data of record.
3. **When you change the generator itself** (`scripts/generate-api.mjs`), commit the script + the regenerated artifacts in the same commit, and add a `HANDOFF.md` entry explaining the schema/shape change. Downstream consumers of `/api/v1/*` will see the diff.
4. CI's `npm run build` does not regenerate artifacts; it does validate them. A PR whose `public/api/v1/*` diff doesn't match its `src/data/*` diff (or its generator diff) should be flagged by review.
5. Artifacts are kept in git (not gitignored) because (a) the diff is reviewable per PR, (b) Vercel gets a deterministic deploy target, and (c) external consumers of `/api/v1/*` benefit from a cacheable, versioned snapshot. The "treat as build artifact and gitignore" alternative was considered and rejected on 2026-05-24.

See [RUNBOOK.md](RUNBOOK.md) §7 for the operational view of this policy.
