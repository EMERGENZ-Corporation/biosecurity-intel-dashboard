# EMERGENZ Dashboard Content Standards
## Source Attribution, Authorship & Data Integrity

**Established on:** Hantavirus Intel Dashboard (2026)  
**Applies to:** All EMERGENZ public-facing intelligence dashboards  
**Version:** 1.0

---

## 1. Source Tier System

Every data point and news item must be assigned a source tier. Tier determines display treatment, disclaimer requirements, and pipeline alert thresholds.

| Tier | Type | Examples | Trust Level |
|---|---|---|---|
| 1 — Authoritative | Official public health agencies | WHO, CDC, ECDC | Highest — pipeline alerts on failure |
| 2 — Institutional | National/regional health agencies, peer-reviewed journals | PHAC, RKI, UKHSA, RIVM, NEJM, Lancet | High — treated as primary source |
| 3 — Media | News outlets, science journalism | Reuters, AP, BBC Health, STAT News | Informational — requires disclaimer |
| 4 — Preprint/Unreviewed | Preprint servers | bioRxiv, medRxiv | Not yet peer-reviewed — must be labeled |

**Rule:** Tier 1 and 2 sources drive structured data fields (case counts, risk levels, clinical guidance). Tier 3 and 4 sources populate news feeds only — never update structured data fields.

## 2. Source Attribution Requirements

### 2.1 Every data point must carry attribution

All structured data fields in `meta.json` (case counts, risk levels, response levels) must have a corresponding `sourceUrl` field linking to the specific authoritative document (e.g., WHO DON, CDC HAN notice).

### 2.2 `sources.json` registry

Every source used in the dashboard must have a registered entry:

```json
{
  "id": "unique-kebab-id",
  "name": "Display Name",
  "url": "https://...",
  "type": "who|cdc|ecdc|national-agency|news|journal|preprint",
  "lastVerified": "YYYY-MM-DD"
}
```

`lastVerified` must be updated whenever the source is manually reviewed. Entries frozen >30 days behind `meta.lastUpdated` during an active outbreak should be flagged for review.

### 2.3 News items require authority field

Every item in `news.json` must carry an `authority` field matching a key in the UI's `AUTHORITY_COLORS` map. Unrecognized authorities render in muted gray — a visual signal that a new source was added to the pipeline without a corresponding UI registration.

### 2.4 News page disclaimer (non-negotiable)

The News Feed page must always display this disclaimer (or equivalent):

> **Note:** News feeds contain media coverage and may include analysis, opinion, or unverified information. For authoritative clinical and public health guidance, see [Protocols & Guidance].

This disclaimer must link to the authoritative guidance section of the dashboard.

## 3. Authorship Standards

### 3.1 Automated pipeline commits

All commits generated by the automated data pipeline must use a named bot identity — never the personal account of a team member:

```
Author: EMERGENZ Data Bot <bot@emergenz.org>
Message: chore(data): automated update YYYY-MM-DDTHH:MM:SSZ
```

### 3.2 Manual commits with AI assistance

All commits where Claude assisted in writing code or content must carry co-authorship attribution:

```
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

### 3.3 `meta.source` field

The `source` field in `meta.json` records which agencies' data informed the current structured case counts. Rules:

- **Never overwritten by the automated pipeline** — the pipeline cannot know which agencies were consulted during manual verification
- Updated only by a human committer when structured data fields are updated
- Format: comma-separated agency names (e.g., `"WHO / ECDC / CDC / BC CDC"`)

### 3.4 Curated content fields

The following fields are always manually authored and must never be set or overwritten by the automated pipeline:

| Field | File | Reason |
|---|---|---|
| `confirmed`, `deaths`, `countries` | `meta.json` | Require verification against official DONs |
| `cdcResponseLevel` | `meta.json` | CDC HAN numbers don't follow extractable patterns |
| `source` | `meta.json` | Tracks which agencies were consulted |
| All entries | `markers.json` | Geographic case markers require human judgment |
| All entries | `us-monitoring.json` | State-level data requires DOH source verification |
| All entries | `ems-briefing.json` | Clinical guidance requires clinical review |
| All entries | `flights.json` | Flight tracing requires human verification |
| All entries | `host-city-biosurveillance.json` | Per-city observations require human source verification and the `publicDisplayAllowed` review gate; per-city status is derived, never hand-set |

## 4. Data Integrity Rules

### 4.1 Never fabricate — use null

When automated extraction (Gemini or equivalent) cannot confidently determine a value, it must return `null`. Guessed or inferred values must never be written to structured data fields.

Gemini prompt rule (must be present in all dashboard pipelines):
> "Never fabricate numbers or events. Only extract what sources explicitly state. Use null for any field you are not confident about."

### 4.2 Null preference over wrong data

A `null` value on the dashboard (shown as `—` or `TBD`) is always preferable to a confidently wrong number. Wrong case counts published during an active outbreak are a patient safety issue.

### 4.3 Integer validation on case counts

Before writing any integer field (confirmed, deaths, countries, usStatesMonitoring), the pipeline must validate:
- Value is a positive integer
- Value is greater than 0
- Value differs from the current stored value

If validation fails, skip the write. Never decrement case counts automatically.

### 4.4 Zero-item pipeline runs

If the RSS pipeline finds zero relevant items in a run:
- Write **no files** — no `meta.json`, no `news.json`, no `timeline.json`
- No git commit
- No Vercel rebuild
- Log the run result to console only

This prevents spurious rebuilds and preserves data integrity when feeds are temporarily quiet.

### 4.5 72-hour sliding window

The pipeline must use a 72-hour sliding window for article deduplication — not a `lastChecked` timestamp. A `lastChecked` cutoff permanently excludes articles missed during feed outages. The sliding window self-heals after any feed failure.

### 4.6 Auto-promoted timeline events (deterministic)

`scripts/promote-news-to-timeline.mjs` is the only automated writer permitted to add events to `src/data/signal-timeline.json`. The contract is:

- **Deterministic only.** No AI is consulted at promotion time. Title and description are verbatim from the source news item — never paraphrased, never summarized, never AI-rewritten. The only permitted transformation is a trailing `…` truncation when the source title exceeds 140 characters (a layout concession; never on the description). The full original title is always recoverable by following the event's `link` field.
- **Strict Tier 1 authority allowlist:** `CDC`, `WHO`, `ECDC`. Items from any other authority — including registered Tier 1 sources whose RSS isn't yet wired into `GLOBAL_FEEDS` (Africa CDC, FDA, etc.) — are skipped.
- **Signal severity gate:** the matched signal's `severity` must be `concern` or `action`. `monitor` and `watch` items skip.
- **Provenance discriminator** is required on every auto-promoted event: `provenance: "auto-news-tier1"`, plus mandatory `newsId`, `authority`, `link`, and `promotedAt` for full traceability back to the source news item.
- **`sourceId` must hard-resolve to a Tier 1 entry in `signal-sources.json`** under the matching authority. If no Tier 1 source is registered for the signal under that authority, the item is skipped — the script never invents a sourceId.
- **No `link`, no promotion.** Items with null or invalid `link` are skipped (attribution requirement, §2.1).
- **Curated wins on collision.** Any existing event (curated OR auto) with the same `signalId` + same UTC day blocks the promotion. The auto-promoter never overwrites human curation.
- **Volume caps:** per-run max 20 promotions; per-signal rolling 7-day max 5 auto events.
- **Zero-promotion runs write nothing** (§4.4 above applies — no file write, no commit).
- **Bot identity:** the workflow step commits as `EMERGENZ Data Bot <bot@emergenz.org>` (§3.1).
- **ID prefix `auto-{newsId}`** is required and validated.

These invariants are enforced by `scripts/validate-data.mjs` and a 12-test unit suite (`scripts/test-promote-news-to-timeline.mjs`) — any code change that weakens them will be caught by `npm run test:validators && npm run test:promote-timeline` in CI.

Existing curated events have no `provenance` field (treated as `"curated"` by default). The UI renders auto and curated events indistinguishably — the underlying authority and attribution are what matters to users; provenance is a maintainer concern surfaced through `automation.dataWriters` in `public/status.json`.

### 4.7 Auto-ingested host-city observations (deterministic)

`scripts/ingest-nwss-host-cities.mjs` is the only automated writer permitted to add observations to `src/data/host-city-biosurveillance.json`. It reads the official CDC NWSS Socrata dataset (`atcp-73re`, "Wastewater Viral Activity Level for SARS-CoV-2, Influenza A and RSV") and writes one observation per US host-city × tracked pathogen for the latest week. The contract:

- **Deterministic only.** No AI. Each value is a CDC categorical activity level (`Very Low`…`Very High`) read verbatim from the official Tier 1 API and mapped through a fixed table. Any unrecognized/missing category is **skipped** — the domain stays "No current data". A missing tile is always preferred over a guessed one (§4.1/§4.2).
- **State-level rollup, faithfully labelled.** NWSS public levels are site-level; the script reports the **highest** category among a state's monitoring sites for the latest week and says exactly that in the observation `summary` (with the site count and week). It does not claim to reproduce CDC's own aggregate.
- **Publish policy — approved once, not per observation.** Because the data is deterministic, official, and Tier 1, re-keying it by hand adds latency, not safety. The default policy is `auto` (`publicDisplayAllowed: true` → live). The `NWSS_PUBLISH_POLICY` env (`auto` | `staged` | `off`) is set explicitly in the workflow so the choice is version-controlled; `staged` routes observations through gate G21 instead, `off` is a kill switch. This is a deliberate, documented exception to the "humans-only writes" default of §3.4 — scoped strictly to deterministic Tier 1 NWSS respiratory data.
- **Severity capped at `watch`.** A single automated wastewater feed can raise a domain to "elevated" and a city to "watch", but never to "concern"/"action". The loudest alarm tiers still require human curation or multi-source convergence (`src/utils/hostCityBioSignals.ts`).
- **Provenance discriminator** `provenance: "auto-nwss"` is required, plus the `auto-nwss-` id prefix, `domain: "respiratory"`, `observationType: "wastewater"`, and `confidence: "official"`.
- **`sourceId` must hard-resolve to a Tier 1 entry** (`cdc-nwss`) in `signal-sources.json`, or the run aborts (config error, not a data gap).
- **Never overwrites human curation.** Each run removes only prior `auto-nwss` observations and rewrites them; it never edits or deletes curated observations, city identity, `sourceIds`, or `sourceCoverageSummary`. US host cities only (Canada/Mexico have no NWSS feed).
- **Self-healing freshness.** If the job silently stops, observations age past the freshness threshold and the UI degrades to "stale" then drops them — stale data is never shown as current. No babysitting required.
- **Fail-open, zero-change no-commit.** Any API/parse error writes nothing (existing data is left intact); a run producing no change makes no commit (§4.4). Bot identity per §3.1.

These invariants are enforced by `scripts/validate-data.mjs` and two unit suites (`scripts/test-validate-data.mjs` auto-nwss cases + `scripts/test-ingest-nwss-host-cities.mjs`) — `npm run test:validators && npm run test:ingest-nwss` in CI catches any weakening.

### 4.8 Auto-ingested host-city observations — Canada / PHAC (deterministic)

`scripts/ingest-phac-host-cities.mjs` is the Canadian counterpart to §4.7, writing `provenance:"auto-phac"` observations for the Canadian host cities (Toronto, Vancouver). It inherits the entire §4.7 contract (deterministic, severity capped at `watch`, self-healing freshness, fail-open, idempotent, never overwrites curated or other-provenance data) with these source-specific points:

- **Source / endpoint.** PHAC National Wastewater Monitoring, Health Infobase **`/src/data/wastewater/wastewater_trend.csv`** (configurable via `PHAC_TREND_URL`). This single trend table is self-contained — each row carries PHAC's categorical `Viral_Activity_Level` **and** a `weekStart` date — so no second fetch is needed. **The legacy `covidLive/wastewater/` path is RETIRED** (it froze at 2024-06) and must not be used for live status. The continuous `min`/`max` metrics are NOT used (thresholding a continuous value into a status = §7.2 human judgement).
- **PHAC is Tier 2**, so `auto-phac` observations resolve `sourceId` to `phac-nwmp` (Tier ≤ 2 permitted by the validator; `auto-nwss` remains Tier 1 only). Provenance id prefix is `auto-phac-`.
- **PHAC's own category is mapped verbatim.** `Viral_Activity_Level` drives status: `High → elevated`; `Moderate`/`Low`/`Non-detect → normal`; `NA2`/blank/unmapped → **skipped**. PHAC's own **city-level aggregate row** (`grouping == "City"`) is used when present; otherwise the highest level among the city's site rows. PHAC's level text is preserved verbatim in the observation `summary`.
- **Pathogens:** `covN2 → SARS-CoV-2`, `fluA → Influenza A`, `rsv → RSV` (parity with NWSS; `fluB` is not surfaced).
- **Observations are dated from `weekStart`** (the data week), not the ingest date.
- **Anti-stale guard (both writers).** If the newest usable `weekStart` is older than `MAX_DATA_AGE_DAYS` (default 45), the observation is **skipped** — the tile shows "No current data" rather than an old level dressed as current. Protects against a dormant feed (and protects NWSS if CDC's feed stalls). Each city is evaluated independently: a current city publishes while a stale one shows "No current data".

Enforced by the same validator and `npm run test:ingest-phac` (which exercises current + stale + malformed CSV fixtures under `scripts/fixtures/`).

### 4.9 Official operational sentinels and signal discovery

`scripts/discover-signals.mjs` is the only automated scanner permitted to create internal `signal-candidates.json` proposals from official signal-discovery lanes. It scans Tier 1/2 epidemiologic sources and official operational sentinels, then reconciles approval-required findings through the reusable `signal-discovery` GitHub issue. It does not use AI and it does not publish clinical guidance.

- **Official epidemiologic sources can support structured candidates.** CDC, FDA, WHO, MOH, and Africa CDC source lanes may support low-risk deterministic monitor/watch candidates when the source is registered in `signal-sources.json` and allowed by `src/data/signal-discovery-policy.json`.
- **Official operational sentinels are bounded.** U.S. Department of State and U.S. Embassy alerts are official government sources for travel, security, operational posture, and cross-border review. They can trigger discovery, watch-level operational alerting, and a human review issue.
- **Operational sentinels cannot write case counts.** Embassy/State-only findings must not overwrite case counts, deaths, disease identity, clinical guidance, or public-health risk levels. Epidemiologic facts require corroboration by WHO, CDC, MOH, FDA, or Africa CDC as appropriate.
- **Auto-published discoveries are capped at `watch`.** Any severity above `watch`, unsupported count change, non-primary claim, clinical text, PPE/isolation/treatment text, or triage-card change must stage for approval.
- **Internal diagnostics stay internal.** Source-lane fetch health and parser diagnostics belong in `signal-candidates.json` and the GitHub issue context, not in public dashboard UI.
- **Every signal carries a refresh classification.** `signals.json` must set `refreshClassification` to one of:
  - `auto-writable` — a deterministic Tier 1/2 source can safely update structured facts, but only inside a tested writer contract and only at `watch` or lower unless a human approves.
  - `auto-checkable-human-reviewed` — automation may check/stage source currency, but structured facts, geography, summaries, markers, and sections remain human-reviewed before publication.
  - `manual-only` — prose/PDF/high-consequence sources require human review before changing structured facts, geography, summaries, markers, sections, or source attribution.
- **Source-currency matrix is prevention, not masking.** The matrix must not hide stale data. A signal clears stale status only after primary-source review, a deterministic writer refresh, or an explicit staged-review finding that leaves the signal stale.

## 5. Public-Facing vs. Internal Information

### 5.1 Never show pipeline diagnostics publicly

The following information is **internal only** and must never appear on the public-facing dashboard:

- Which RSS feeds failed on the last run
- Pipeline run timestamps (distinct from data `lastUpdated`)
- Gemini API errors or retry counts
- Bright Data / enrichment failures
- `feedHealth` object contents

### 5.2 Acceptable public transparency

The following pipeline state information is appropriate for public display:

| Element | Rationale |
|---|---|
| Data `lastUpdated` timestamp | Users need to know data currency |
| Pipeline update interval ("Auto-updated every 6h") | Sets user expectations |
| Staleness warning (>48h, active outbreak only) | User-relevant data quality signal |
| Source chips on data points | Attribution transparency |

### 5.3 Staleness warning threshold

Display a staleness warning when `Date.now() - lastUpdated > 48 hours` **AND** the outbreak is active (confirmed > 0). Do not show during resolved/inactive periods.

## 6. Authoritative Feed Failure Protocol

### 6.1 Tier 1 feed failure = hard alert

If WHO, CDC, or ECDC RSS feeds fail during an active outbreak (confirmed > 0):
- Script must exit with code 1
- GitHub Actions marks the run as failed
- Pipeline failure issue is auto-created in the repository
- Repository owner receives email notification

### 6.2 Tier 2+ feed failure = soft warning

Non-authoritative feed failures (news outlets, journals, national agencies) are:
- Logged to console
- Tracked in `feedHealth.failedFeeds[]`
- Never surfaced publicly
- Do not cause the pipeline to exit with error

### 6.3 WHO RSS URL monitoring

WHO RSS feed URLs have historically changed without notice (documented 404 in May 2026). All Tier 1 feed URLs should be manually verified when a pipeline run returns 0 WHO items despite an active outbreak.

## 7. Clinical and Public Health Content Standards

### 7.1 Clinical guidance is manually curated only

The EMS Briefing, Protocols & Guidance section, and any clinical recommendations must be:
- Written or reviewed by a qualified person
- Never auto-generated by the pipeline
- Sourced to specific guidelines, DON pages, or clinical publications
- Timestamped with a `lastReviewed` or `updatedAt` field

### 7.2 CFR and epidemiological calculations

Case fatality rate and other epidemiological calculations displayed on the dashboard must:
- Be computed from verified confirmed and death counts only
- Never be extracted from media coverage
- Be clearly labeled as preliminary during active outbreaks

### 7.3 Risk level sourcing

Risk level fields (`whoGlobalRisk`, `ecdcRisk`, `cdcResponseLevel`) must link directly to the source document — not to a general agency homepage. Example: link to the specific WHO DON, not `who.int`.

## 8. Commit and Release Standards

### 8.1 Conventional commit format

All commits must follow conventional commit format:

```
type(scope): short description

Body (optional)

Co-Authored-By: (if AI-assisted)
```

Types: `feat`, `fix`, `chore`, `data`, `refactor`, `docs`

### 8.2 Tagged restore points

Before any major data or pipeline change, tag a restore point:

```bash
git tag -a stable-<descriptor> <commit> -m "<description>"
git push origin <tag>
```

### 8.3 Revert procedure

**Safe (preferred):**
```bash
git revert <commit> --no-edit && git push origin main
```

**Hard rollback:**
```bash
git reset --hard <tag-or-commit> && git push origin main --force
```
